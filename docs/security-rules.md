# Security rules reference

Every detection rule the product ships, in one queryable registry
(`src/core/rules/registry.ts`). This page is generated by reading that registry,
not hand-typed — see "How this table was generated" at the bottom, and re-run
that command if the registry ever changes.

## Namespace: `AAA-*` (current) vs `AAC-*` (legacy)

Rules were renamed from `AAC-*` (Aster Agent **Console**) to `AAA-*` (Aster Agent
**Audit**) as part of the product rename; the domain/number part is stable
(`AAC-GIT-014` → `AAA-GIT-014`). Both ids keep working everywhere a rule id is
accepted:

- Findings **already persisted** under the old id keep that id — nothing is
  rewritten in the database.
- `policy.json` (`ignoreRules`, `rules`) accepts either generation — see
  [docs/policies.md](policies.md).
- `resolveRule(id)` looks a rule up by its current id **or** any legacy id, so
  every consumer (policy, SARIF export, HTML report, CLI) treats the two
  generations as the same rule.

## Confidence and detection method

Two metadata fields describe *how much to trust* a match, independent of severity:

- **`confidence`** — `"high"` for structural facts about on-disk configuration
  (a JSON/TOML config either declares a dangerous pattern or it doesn't);
  `"medium"` for regex matches over free-text shell commands (real but has false
  positives/negatives — see [docs/limitations.md](limitations.md)).
- **`detectionMethod`** — `"runtime-event"` (observed in an ingested agent tool
  call) vs `"static-config"` (found by scanning an MCP config file on disk,
  never executed).

`references` (CWE / OWASP-LLM / OWASP-Agentic / MCP-best-practice mappings) is
part of every rule's shape but **ships empty for all 21 rules today**. The
registry's own header comment is explicit about why: "a wrong mapping is worse
than none," and curated mappings are future work, not a current claim. Standards
coverage described anywhere else for this tool should be read as **structure
exists, mapping does not yet.**

Counts by source, so the total is auditable: 9 rules from `riskRuleCatalog()`
(shell/git), 3 inline runtime rules defined directly in `src/core/rules/registry.ts`
(`AAA-SECRET-001`, `AAA-SECRET-004`, `AAA-FILE-005` — these have context-dependent
severity at detection time; the table shows the highest severity each can assign),
and 9 from `mcpRuleCatalog()` (the MCP static-config rules, detailed in

The registry **derives** these entries from the same live rule tables that
actually run detection (`riskRuleCatalog`, `mcpRuleCatalog`) rather than
duplicating titles/severities by hand — so this table cannot drift from the code
that detects. Detection logic itself lives in `src/core/risk.ts` (runtime) and
`src/core/mcp.ts` (static-config); the registry in `src/core/rules/registry.ts`
is metadata and id-mapping only, not a second detection engine.

## Rule versioning

Every rule carries a `version` field (currently `"1.0.0"` for all 21). Per the
registry's header comment, this is bumped when a rule's detection logic or
default severity changes — a way to tell, in an exported report or evidence
bundle, whether a finding was produced by the same rule semantics as today's.

import { ruleRegistry } from './src/core/rules/registry.ts';
const rules = ruleRegistry();
console.log('TOTAL', rules.length);
for (const r of rules) {
  console.log([r.id, r.legacyIds.join('|'), r.category, r.defaultSeverity, r.confidence, r.detectionMethod, JSON.stringify(r.title)].join('\t'));
}
"
